Spring Boot过滤器Filter从入门到实战

Zane Wang2026-07-082026-07-08

当用户在浏览器点击一个按钮，浏览器会向服务器发送一个 HTTP 请求，这个请求在到达你写的业务代码（Controller）之前，会先经过一道「关卡」，这道关卡就是过滤器。过滤器可以检查这个请求是否合法，比如用户有没有登录、请求参数有没有乱码、请求频率是不是太高等等。检查通过就放行，让请求继续往后走；检查不通过就拦截，直接返回错误信息，请求根本到不了 Controller。

1. 过滤器的执行流程全景图

要真正用好过滤器，必须搞懂它的执行流程。咱们用一个完整的请求链路来讲解。假设用户在浏览器点击了一个按钮，发起了一个 HTTP 请求，这个请求从到达服务器到返回响应，会经历下面这条完整的链路。

第一步，浏览器发送 HTTP 请求到服务器，服务器（如 Tomcat）接收到请求。第二步，请求进入过滤器链（FilterChain），如果有多个过滤器，会按照配置的顺序依次执行每个过滤器的请求处理逻辑。第三步，所有过滤器都放行后，请求到达 Servlet（在 Spring Boot 中是 DispatcherServlet），Servlet 将请求分发给对应的 Controller 方法。第四步，Controller 方法执行完毕，生成响应结果。第五步，响应沿原路返回，再次经过过滤器链，这次执行的是每个过滤器的响应处理逻辑（顺序与请求时相反）。第六步，响应返回给浏览器，整个流程结束。

2. 放行、拦截、终止：过滤器的三种核心动作

过滤器的核心能力可以归纳为三种动作：放行、拦截、终止。理解这三种动作的区别和写法，是掌握过滤器的关键。下面逐一讲解。

2.1 放行：让请求继续往后走

放行是过滤器最常用的动作。当检查通过，需要让请求继续传递给下一个过滤器或 Servlet 时，就调用 FilterChain 的 doFilter 方法。这一行代码是「放行」的标志，写不写它直接决定了请求能不能往后走。下面是放行的标准写法。

2.2 拦截：不让请求往后走

拦截就是不让请求继续往后走，直接在过滤器里返回响应。实现方式很简单：不调用 chain.doFilter，而是直接用 response 写入响应内容并返回。比如用户未登录时，直接返回一段 JSON 提示「请先登录」，请求根本到不了 Controller。下面是拦截的典型写法。

2.3 终止：异常情况下的中断

终止和拦截类似，但更强调「异常中断」。比如在请求前处理阶段抛出异常，或者在检测到非法请求时立即终止处理。终止本质上也是不调用 chain.doFilter，区别在于触发原因不同——拦截通常是业务规则不满足（如未登录），终止通常是异常或错误（如参数解析失败）。写法上，终止往往配合异常处理或直接返回错误响应。
