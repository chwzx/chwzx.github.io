Spring Boot 拦截器从入门到实战

如果你刚学完 Spring Boot 的 Controller、Service、Mapper 三层架构，能写出增删改查接口，那恭喜你已经入门了。但某天你接到一个需求：所有接口都要校验用户是否登录，没登录的请求直接打回去。你会怎么做？最笨的办法是每个 Controller 方法开头都写一段 if (user == null) return “未登录”，几十个接口写下来人都要麻了。这时候，拦截器就是你的救星。

拦截器（Interceptor）是 Spring MVC 提供的一种机制，能让你在请求到达 Controller 之前、Controller 处理完之后、响应返回之前这三个时机，统一插入一段逻辑。登录校验、权限拦截、请求日志、接口防刷这些横切多个接口的公共需求，用拦截器实现最合适。它是 Java Web 开发里和过滤器（Filter）齐名的两大拦截体系之一，也是新手从"会写接口"进阶到"会做工程"的必修课。

1. 拦截器的执行时机

拦截器有三个执行时机，分别对应三个核心方法。记住这三个时机，就掌握了拦截器的核心：

  1. preHandle：在 Controller方法执行之前执行。这是最常用的时机，登录校验、权限拦截都在这里做。返回 true 表示放行，请求继续往下走；返回 false 表示拦截，请求到此结束，到不了 Controller。
  2. postHandle：在 Controller 方法执行完之后、视图渲染之前执行。注意是 Controller 正常返回才会执行，如果 Controller 抛异常则不会执行。可以在这里修改 ModelAndView，但前后端分离项目基本用不到。
  3. afterCompletion：在请求完全结束之后执行（视图渲染完成、响应已返回）。无论 Controller 是否抛异常，这个方法都会执行（类似 try-finally 的 finally）。常用于资源清理、记录最终耗时。

2. 三大核心方法详解

三个方法都有 default 实现（JDK 8+ 特性），所以你实现 HandlerInterceptor 接口时，不用三个都重写，按需重写就行。实际开发中，preHandle 用得最多，afterCompletion 次之（用于清理或记耗时），postHandle 在前后端分离项目里基本不用。

参数里的 handler 是什么？它其实是即将要执行的 Controller 方法的包装对象（HandlerMethod 类型），通过它能拿到 Controller 类、方法名、方法参数、方法上的注解等信息。比如你想知道当前请求要执行哪个 Controller 的哪个方法，可以强转成 HandlerMethod 然后取方法名。

5. 多个拦截器的执行顺序

实际项目里往往不止一个拦截器。比如登录拦截器、权限拦截器、日志拦截器同时存在。它们的执行顺序怎么控制？答案是用 order 方法指定优先级，数字越小优先级越高（越先执行）。

6. 拦截器无法获取请求体的坑

这是新手必踩的坑。如果你想在拦截器里读取请求体（比如校验请求参数），会发现读了一次后 Controller 里就读不到了。原因是请求流（InputStream）只能读一次，读完后流就关闭了。

解决方案是用 HttpServletRequestWrapper 包装请求，把请求体缓存起来，可以反复读。这个稍微复杂，新手了解原理即可，实际项目可以用 Spring 提供的 ContentCachingRequestWrapper：

7. 拦截器里获取 Controller 方法信息

在 Spring MVC 的拦截器中，preHandle 方法的第三个参数 Object handler 就是我们要找的关键。如果这个请求是由一个 @Controller 或 @RestController 的方法处理的，那么这个 handler 实际上是 HandlerMethod 类型的对象。通过将 handler 强转为 HandlerMethod，我们就可以拿到即将执行的 Controller 方法的一切信息（方法名、参数、注解等）。
